## 2026-03-03

### BUG-005：区间符 `~` 在 pandoc 转 Word 时变为下标小字

**现象**：MD 转 Word 后，生命周期范围文本（如 `90~120天段`、`0~7天`）中两个 `~` 之间的文字字号异常缩小，变为下标格式。

**根因**：pandoc 将 `~text~` 解析为 HTML `<sub>text</sub>` 下标标记。若范围写作 `90~120天段`，pandoc 将 `~120天段~` 后一个 `~` 与前面的 `~` 配对，导致 `120天段` 全部变为下标。

**影响范围**：所有使用 `~` 作为区间连接符的章节文本（付费用户生命周期分组：0~7天、8~30天、31~90天、90~120天段、120~365天段、365天+）。

**修复**：
- 将月报草稿 `wip_report.md` 中全部 `~` 连接符替换为 `-`（如 `0-7天`、`90-120天段`）
- `style_guide.json` → `general_rules` 追加禁用规则
- `SKILL.md` → 第1-D节写作规范追加 `~` 禁用说明

**状态**：✅ 已修复并写入规范

### BUG-006：内购章节 Top3 点位排版过密，阅读困难

**现象**：第5章付费礼包监控的 Top3 点位以逗号分隔写在同一行，多个点位数据挤在一起，Word 中可读性差。

**根因**：生成时使用了行内列举格式，未考虑每个点位有多项指标（收入、ARPPU、用户数、变化率）的信息密度。

**修复**：将 Top3 点位从行内列举改为子 bullet list，每个点位独占一行：
```markdown
2. **Top3 点位（按收入排名）：**
   - **PIGGY_BANK $3,307（+62%）**：...
   - **DIAMOND_STORE $2,106（-3%）**：...
   - **BANKRUPT_SALE $1,550（+70%）**：...
```

**状态**：✅ 已修复，`style_guide.json` 中 `iap_monitor.format` 已有子 bullet 说明，无需额外规范更新

## 历史 Bug（脚本开发阶段）

### BUG-001：parse_resource.py DictReader 重复列名导致环比全为 0%

**现象**：付费用户资源监控章节（钻石消耗/发放）所有指标环比变化均显示为 0%。

**根因**：CSV 文件中存在重复列名（如两列都叫"金币"）。Python `csv.DictReader` 遇到重复列名时，后者覆盖前者，导致本月/上月数据被错误读取为同一值，环比计算结果为 0。

**修复**：在 `parse_resource.py` 中读取 CSV 时，对重复列名追加序号后缀（`金币_1`、`金币_2`）以区分。

**状态**：✅ 已修复

---

### BUG-002：基础数据(其1) 日期列含星期后缀无法解析

**现象**：`parse_basic.py` 读取 `基础数据(其1)` 时报 `ValueError: time data '2026-01-01(四)' does not match format`，脚本崩溃。

**根因**：看板导出的日期列格式为 `YYYY-MM-DD(一/二/三/四/五/六/日)`，带星期汉字后缀，Python `datetime.strptime` 默认格式无法解析。

**修复**：在解析日期前，用正则 `re.sub(r'\(.\)$', '', date_str)` 去除括号内星期字符，再按 `%Y-%m-%d` 解析。

**状态**：✅ 已修复

---

### BUG-003：基础数据(其1) 日期列名不统一（"日期" vs "时间"）

**现象**：`parse_basic.py` 按列名 `"日期"` 查找日期列时 `KeyError`，脚本崩溃。

**根因**：看板不同版本/导出方式下，日期列名称可能为 `"日期"` 或 `"时间"`，不固定。

**修复**：用 `next((col for col in header if col in ('日期', '时间')), None)` 兼容两种列名，若均不存在则降级使用第一列。

**状态**：✅ 已修复

### BUG-004：Windows 环境下 Bash 管道命令解析失败

**现象**：在 Windows bash 中运行带 `|` 管道的 Python 命令（如 `python script.py | python next.py`）时，子命令参数解析异常或输出乱码。

**根因**：Windows 下 Git Bash 对多级管道的 Python 子进程标准输出编码处理与 Linux 不同，且部分特殊字符（中文路径、括号）在管道中传递时丢失。

**修复**：改用 `--output <文件路径>` 参数写入中间 JSON 文件，避免管道传递，各步骤独立读取文件。`run_all.py` 所有步骤均使用文件输出而非 stdout 管道。

**状态**：✅ 已修复

---

### BUG-004b：文件名含括号时 glob 匹配失败

**现象**：`基础数据(其1)` 文件名含 `(` `)` 括号，`glob.glob("*基础数据(*")` 匹配失败，返回空列表。

**根因**：Python `glob` 将 `[` `]` `(` `)` 视为特殊字符（bracket expression），直接传入含括号的模式时行为不一致（部分版本需转义）。

**修复**：改用 `os.listdir()` 遍历目录后手动用 `fnmatch` 或字符串 `in` 检查文件名，或对 glob 模式中的括号用 `[[]` `[]]` 转义。

**状态**：✅ 已修复

### BUG-008：低质量草稿可被直接归档，表现为“像是直接 copy”

**现象**：当草稿质量不达标时，最终文件仍可能直接由 `wip_daily_report.md` 复制归档，导致报告质量波动明显。

**根因**：当前流程是“写草稿 -> 复制归档”，缺少强制性的归档前质量门；`--strict` 只校验 CSV 覆盖和解析，不校验文稿质量。

**修复**：
- 本次先落地流程修复：归档前执行人工质量门（章节覆盖、双周期、生命周期拆档、风险动作完整性）
- 已在 `CHANGE_LOG.md` 固化 SOP

**状态**：🔄 进行中（流程门禁已执行，脚本化门禁待补）
